# Coding Rules

## ゲッター・セッターは、なるべく利用しない。

### 意図
ゲッター、セッターが安易に存在することで、情報隠蔽が適切に行えない「貧血症ドメインモデル（Anemic Domain Model）」というバッドプラクティスに陥りやすくなります。本来、クラスはなるべく「振る舞い」を公開するように設計すべきです。

しかし、ゲッター・セッターを用いる方が、シンプルに実装できるケースも存在します。
複雑さが解消されるかどうかを考慮し、適切に判断してください。

とはいえ、そもそもクラス設計を見直すことで、ゲッター・セッターを不要にできる場合も多いです。
まずは、クラス設計を見直すことを検討してください。

## UIに関係しないロジックはapp/modulesディレクトリに配置する

### 意図
UIとロジックを分離することで、下記の利点を得るためです。
- UIの変更がロジックに影響を与えないようになる。
- UIを考慮せずにロジックが設計できるため、コードの可読性・保守性が向上する。
- ユニットテストが書きやすくなる。
